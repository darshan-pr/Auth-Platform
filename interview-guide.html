рҹ”җ 1. OAuth 2.0 & PKCE Flow

Q1: Walk me through the OAuth 2.0 Authorization Code Flow with PKCE as implemented in your project.

PROJECTв–ј

Answer:

  1. Client redirects to GET /oauth/authorize with client_id, redirect_uri, state, code_challenge, code_challenge_method=S256.
  2. Server validates вҖ” checks the app exists, redirect_uri is registered (or localhost in dev), PKCE challenge is present, state is present. Creates an OAuth session in Redis with a session_id.
  3. Renders the hosted login page (Jinja2 template) вҖ” the client app never sees user credentials.
  4. User submits credentials via POST /oauth/authenticate. On success, the server generates an authorization code, stores it in Redis with the code_challenge & user info, and returns a redirect URL back to the client's redirect_uri?code=...&state=....

рҹ”Қ How to dig deeper

  • Why is PKCE mandatory? вҶ’ Prevents authorization code interception attacks for public clients (SPAs, mobile apps) that can't securely store a client_secret.
  • Why S256 and not plain? вҶ’ Plain method sends verifier = challenge, making interception trivial. S256 is one-way hash.
  • Ask about the consent screen flow вҖ” the project has a dedicated /oauth/consent endpoint.

Q2: Why does your project store OAuth sessions in Redis instead of the database?

DESIGNв–ј

Answer: OAuth sessions are ephemeral вҖ” they live only for the duration of the authorization flow (typically <5 minutes). Storing them in Redis provides:

  • Automatic expiry via TTL вҖ” no cleanup cron needed
  • Sub-millisecond reads during the code exchange, critical for user-facing latency
  • No schema migrations needed for session structure changes
  • Atomicity вҖ” Redis SETEX is atomic; no transaction overhead

The database is reserved for durable state: users, apps, tenants, consent records, login events.

рҹ”Қ How to dig deeper

Ask: "What happens if Redis goes down mid-flow?" вҶ’ The session is lost; user must restart. This is acceptable because the flow is short-lived. For critical data (e.g., user records), PostgreSQL provides durability.

Q3: Explain the Silent SSO mechanism in your /oauth/authorize endpoint.

PROJECTв–ј

Answer: When a user has already authenticated, a platform_sso HttpOnly cookie is set. On subsequent /oauth/authorize calls:

  1. Server reads the platform_sso cookie and verifies the JWT inside it.
  2. Checks the type == "platform_sso" and that the tenant_id matches the requesting app.
  3. If valid вҶ’ skips the login page entirely, binds the OAuth session to the existing user, and redirects straight to the consent page.
  4. The prompt parameter can override this: prompt=login forces re-authentication, prompt=consent forces the consent screen.

This mirrors how Google/Okta SSO works вҖ” once signed in, other apps under the same tenant get seamless access.

рҹӘҷ 2. JWT, RS256 & Key Management

Q4: Why RS256 over HS256 for JWTs? What are the trade-offs?

COREв–ј

Answer:

AspectHS256 (Symmetric)RS256 (Asymmetric)
KeyShared secretPrivate key signs, public key verifies
Who can verify?Only holders of the secretAnyone with the public key
Microservice fitBad вҖ” every service needs the secretGreat вҖ” distribute public key, keep private key safe
PerformanceFaster (HMAC)Slower (RSA math)

In this project, RS256 is critical because client apps must verify tokens without knowing the signing secret. The auth server signs with the private key; any microservice or client can verify with the public key via /.well-known/jwks.json.

Q5: Walk me through the key management strategy in jwt_service.py.

PROJECTв–ј

Answer: The project uses a 3-tier fallback strategy:

  1. Environment variables first вҖ” JWT_PRIVATE_KEY_PEM and JWT_PUBLIC_KEY_PEM (ideal for cloud deployments like Railway)
  2. Disk files second вҖ” reads from backend/keys/private_key.pem and public_key.pem
  3. Auto-generate last вҖ” generates a 2048-bit RSA key pair and persists to disk. In production with REQUIRE_PERSISTENT_JWT_KEYS=true, this raises an error instead вҖ” preventing accidental key rotation on redeployment.
# From jwt_service.py вҖ” the fallback chain:
def _load_or_generate_keys():
    env_keys = _load_keys_from_env()     # Priority 1
    if env_keys: return env_keys
    disk_keys = _load_keys_from_disk()   # Priority 2
    if disk_keys: return disk_keys
    if settings.IS_PRODUCTION and settings.REQUIRE_PERSISTENT_JWT_KEYS:
        raise RuntimeError("No JWT key material found")  # Fail-safe
    # Priority 3: generate new keys
    private_key = rsa.generate_private_key(65537, 2048)
    _persist_keys(private_key, private_key.public_key())
    return private_key, private_key.public_key()

рҹ”Қ How to dig deeper

Ask: "What happens if keys rotate?" вҶ’ All existing JWTs become invalid because the public key changes. Solutions: Key ID (kid) in JWT header + JWKS endpoint serving multiple keys during rotation window.

Q6: How does your project handle token revocation without a token blacklist table?

PROJECTв–ј

Answer: The project uses a user-level blacklist in Redis rather than a per-token blacklist:

# When admin force-logs-out a user:
redis_client.delete(f"user_online:{tenant_id}:{user_id}")
redis_client.setex(f"user_blacklist:{tenant_id}:{user_id}", 86400, "1")
redis_client.publish(f"force_logout:{tenant_id}:{user_id}", "revoked")

# During token verification:
def verify_token(token):
    payload = jwt.decode(token, PUBLIC_KEY, ...)
    if is_user_blacklisted(payload["user_id"], payload["tenant_id"]):
        return None  # Reject even though token is cryptographically valid

Why user-level? A per-token blacklist grows unboundedly and requires checking every request. User-level blacklisting is O(1) lookup, automatically expires after 24 hours, and covers all tokens for that user at once вҖ” perfect for force-logout scenarios.

The blacklist is cleared on next successful login via clear_user_blacklist().

рҹ“Һ 3. DPoP вҖ” Sender-Constrained Tokens

Q7: What problem does DPoP solve that bearer tokens don't?

COREв–ј

Answer: Bearer tokens are "whoever holds it, owns it". If stolen (via XSS, logs, or network interception), the attacker can use them freely. DPoP (RFC 9449) solves this by binding the token to a client's cryptographic key pair:

  1. Client generates an ephemeral key pair
  2. On each request, client signs a proof JWT containing the HTTP method, URL, and timestamp with its private key
  3. Server verifies the proof's signature and checks that the JWK thumbprint matches the cnf.jkt claim embedded in the access token

Result: A stolen access token is useless without the client's private key. This is called a sender-constrained token.

рҹ”Қ How to dig deeper

  • How is replay prevented? вҶ’ jti (unique nonce) stored in Redis with 5-min TTL
  • How is the binding established? вҶ’ During /oauth/token, if a DPoP header is present, server computes JWK thumbprint and embeds it as cnf.jkt in the access token
  • What if the DPoP header is missing on a bound token? вҶ’ The resource server should reject it

Q8: What specific validations does your DPoP service perform?

PROJECTв–ј

Answer: From dpop_service.py, validate_dpop_proof() checks:

  1. Structure вҖ” JWS must have exactly 3 parts (header.payload.signature)
  2. Header typ == "dpop+jwt"
  3. Algorithm вҖ” must be asymmetric. Rejects none, HS256, HS384, HS512
  4. JWK present вҖ” public key in header, and must NOT contain "d" (private key material)
  5. Cryptographic signature вҖ” reconstructs public key from JWK and verifies the JWT signature using PyJWT
  6. htm matches the actual HTTP method
  7. htu matches the actual request URI
  8. iat within 5 minutes of current server time
  9. jti present and not replayed (checked via Redis with 5-min TTL)
  10. ath (if access_token provided) вҖ” SHA-256 of the access token must match

рҹ”‘ 4. WebAuthn / Passkeys

Q9: Explain the WebAuthn registration and authentication flows as you implemented them.

PROJECTв–ј

Registration:

  1. Server calls generate_passkey_registration_challenge() вҖ” creates 32 random bytes, stores in Redis with 5-min TTL, returns PublicKeyCredentialCreationOptions
  2. Client calls navigator.credentials.create() and sends back the attestationObject + clientDataJSON
  3. Server verifies: challenge matches, type == "webauthn.create", RP ID hash matches, user-present flag is set. Extracts the COSE public key from the attestation's authData.
  4. Stores credential_id, public_key, sign_count, algorithm in the database.

Authentication:

  1. Server sends challenge + allowCredentials list вҶ’ client calls navigator.credentials.get()
  2. Client returns authenticatorData, clientDataJSON, signature
  3. Server verifies: challenge exists in Redis, RP ID hash, flags (user present + verified), sign count incremented (cloned authenticator detection), and cryptographic signature over authData || SHA-256(clientDataJSON)

рҹ”Қ How to dig deeper

  • Why does your project parse CBOR manually? вҶ’ To avoid external CBOR library dependencies. The _extract_auth_data_from_cbor() function does a simple byte-scan for the "authData" key.
  • What's the sign count check? вҶ’ Each authenticator increments a counter. If the server sees a non-increasing count, it likely meansвҖ”the key was cloned.

рҹҸў 5. Multi-Tenancy & Data Isolation

Q10: How does your project achieve multi-tenant data isolation?

PROJECTв–ј

Answer: The project uses shared-database, discriminator-column isolation:

  • Every data model has a tenant_id foreign key pointing to the tenants table
  • The User model has a composite unique constraint: UniqueConstraint('email', 'tenant_id', 'app_id') вҖ” same email can exist in different tenants
  • Every query filters by tenant_id: db.query(User).filter(User.tenant_id == app.tenant_id, ...)
  • Tokens embed tenant_id in their payload, so verification is tenant-scoped
  • Redis keys are namespaced: user_online:{tenant_id}:{user_id}, user_blacklist:{tenant_id}:{user_id}

рҹ”Қ How to dig deeper

Ask about alternatives: Schema-per-tenant (better isolation, harder ops) vs. Database-per-tenant (strongest isolation, hardest to manage). The shared-table approach works well for <10K tenants where cross-tenant queries and simpler operations are priorities.

Q11: What's the relationship between Tenant, App, and User in your data model?

PROJECTв–ј

Answer:

Tenant (org/company) в”ңв”Җв”Җ App A (client_id, redirect_uris, OTP settings, token TTLs) в”Ӯ в”ңв”Җв”Җ User 1 (email: alice@x.com, tenant_id, app_id) в”Ӯ в””в”Җв”Җ User 2 (email: bob@x.com, tenant_id, app_id) в”ңв”Җв”Җ App B в”Ӯ в””в”Җв”Җ User 3 (email: alice@x.com вҶҗ same email, different app!) в””в”Җв”Җ Admin (manages all apps in this tenant)

рҹӣЎпёҸ 6. Security Hardening

Q12: Explain the CSRF protection strategy in your project.

SECURITYв–ј

Answer: The project uses the Double-Submit Cookie Pattern in csrf.py:

  1. On any GET request, if no csrf_token cookie exists, the middleware sets one with secrets.token_urlsafe(32)
  2. On POST/PUT/DELETE requests, the middleware checks that X-CSRF-Token header == csrf_token cookie using hmac.compare_digest() (timing-safe comparison)
  3. Exemptions: Requests with Bearer auth header (API-to-API), application/json content-type (CORS preflight already prevents cross-origin JSON POST), /oauth/token (protected by PKCE), and docs endpoints

The cookie is httponly=False so JavaScript can read it and include it in headers вҖ” this is by design for the double-submit pattern.

Q13: How does your brute-force protection work?

PROJECTв–ј

Answer: Two layers in auth.py:

Layer 1 вҖ” Per-Account Lockout:

  • Redis counter login_attempts:{app_id}:{email} is incremented on each failed login
  • After 5 failures вҶ’ a login_lockout:{app_id}:{email} key is set with 15-min TTL
  • Before every login attempt, _check_account_lockout() checks the lockout key and returns HTTP 429 with remaining time
  • On successful login вҶ’ _clear_login_attempts() deletes both keys

Layer 2 вҖ” Rate Limiting:

  • create_rate_limit_dependency() creates FastAPI dependencies that enforce per-IP request limits: login=5/min, signup=10/min, OTP=3/min

Fail-open design: If Redis is down, both layers silently skip (fail open) вҖ” availability over strict security, with the tradeoff documented.

Q14: Your forgot-password endpoint returns success even for non-existent users. Why?

SECURITYв–ј

Answer: This prevents email enumeration attacks. If the server returned "user not found" for invalid emails and "OTP sent" for valid ones, an attacker could probe the system to discover which emails are registered.

# From auth.py:
if not user:
    # Don't reveal if user exists вҖ” always return success
    return ForgotPasswordResponse(
        message="If an account with this email exists, a password reset has been sent.",
        ...
    )

This is an OWASP best practice. The response is intentionally identical regardless of whether the user exists.

рҹҸ—пёҸ 7. System Design & Architecture

Q15: If this auth platform needs to serve 10 million users, what would you change?

DESIGNв–ј
  • Database вҖ” Shard tenants across multiple PostgreSQL instances (tenant-based partitioning). Add read replicas for token verification queries.
  • Redis вҖ” Move to Redis Cluster for horizontal scaling. Session data is already key-value, so it shards naturally.
  • Key rotation вҖ” Implement JWKS endpoint with kid (key ID) headers. Support overlapping key windows during rotation.
  • Rate limiting вҖ” Move to a distributed rate limiter (e.g., sliding window with Redis Lua scripts or token bucket).
  • Stateless sessions вҖ” Current Redis sessions are fine up to millions, but could add a stateless JWT-based session fallback.
  • Geographic distribution вҖ” Deploy auth servers in multiple regions with cross-region PostgreSQL replication.

Q16: Why did you choose FastAPI over Django/Flask for this project?

DESIGNв–ј
  • Async support вҖ” FastAPI is ASGI-native. Auth flows involve Redis lookups, DB queries, email sending вҖ” async I/O is a natural fit.
  • Automatic OpenAPI docs вҖ” critical for an auth platform where client developers need clear API contracts
  • Pydantic validation вҖ” request/response models like AuthenticateRequest with typed fields are validated automatically
  • Dependency injection вҖ” Depends(get_db), Depends(_rl_login) make middleware composable and testable
  • Performance вҖ” FastAPI on Uvicorn benchmarks 5-10x faster than Flask for I/O-bound workloads

Q17: Explain the middleware execution order in your app and why it matters.

PROJECTв–ј 
# From main.py вҖ” middleware added in this order:
app.add_middleware(CORSMiddleware, ...)   # Added first
app.add_middleware(ConsoleAuthMiddleware) # Added second
app.add_middleware(CSRFMiddleware)        # Added third

Starlette processes middleware in LIFO order (last added = first executed). So the actual request flow is:

  1. CSRFMiddleware вҶ’ validates CSRF tokens on state-changing requests
  2. ConsoleAuthMiddleware вҶ’ protects /dashboard routes, redirects to login
  3. CORSMiddleware вҶ’ handles preflight OPTIONS and adds CORS headers
  4. Route handler executes

This order ensures CSRF is checked before auth logic runs, and CORS headers are always added even on blocked requests.

рҹ—„пёҸ 8. Database & Redis Patterns

Q18: Explain the SQLAlchemy session management pattern in your project.

COREв–ј 
# db.py вҖ” Generator-based dependency:
def get_db() -> Generator:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

# Used as a FastAPI dependency:
@router.post("/login")
def login(db: Session = Depends(get_db)):
    ...

This is the Unit of Work pattern: each request gets its own session, which is guaranteed to close even on exceptions via finally. FastAPI's dependency injection ensures the generator is properly exhausted after the request completes. No connection leaks.

рҹ”Қ How to dig deeper

Ask about autoflush=False: prevents SQLAlchemy from auto-flushing pending changes before queries, giving explicit control over when writes happen. This is important in auth flows where you want to validate before committing.

Q19: How does your project use Redis pipeline/mget for online status?

PROJECTв–ј 
# jwt_service.py вҖ” Batch online status check:
def get_online_status_map(user_ids, tenant_id):
    keys = [f"user_online:{tenant_id}:{uid}" for uid in user_ids]
    mget = getattr(redis_client, "mget", None)
    if callable(mget):
        values = mget(keys)  # Single round-trip for N keys!
        return {uid: bool(value) for uid, value in zip(user_ids, values)}
    # Fallback: individual EXISTS calls
    return {uid: redis_client.exists(key) == 1 for uid, key in zip(user_ids, keys)}

Why MGET? The admin dashboard shows online status for potentially hundreds of users. Without MGET, that's N round-trips to Redis. With MGET, it's 1 round-trip regardless of N. The fallback exists for Redis versions/clients that don't support MGET.

рҹҗҚ 9. Python / FastAPI Internals

Q20: What is the GIL and how does it affect your auth server?

COREв–ј

The Global Interpreter Lock allows only one thread to execute Python bytecode at a time. For an auth server, this matters because:

  • CPU-bound work (RSA signing, bcrypt hashing) вҖ” GIL serializes these. The project mitigates this by using Gunicorn with multiple workers in production (each worker is a separate process with its own GIL).
  • I/O-bound work (Redis, PostgreSQL, SMTP) вҖ” GIL is released during I/O system calls, so async FastAPI handlers can serve thousands of concurrent requests efficiently.

The Procfile uses gunicorn with uvicorn.workers.UvicornWorker вҖ” multiple processes Г— async I/O = maximum throughput.

Q21: Your validate_app_credentials uses hmac.compare_digest. Why not just ==?

SECURITYв–ј

== is a short-circuit comparison вҖ” it returns False as soon as the first mismatched character is found. An attacker can measure response times at microsecond precision to guess the secret one character at a time (timing side-channel attack).

hmac.compare_digest() is constant-time вҖ” it always compares all bytes regardless of where the mismatch occurs, making timing analysis impossible.

# Vulnerable:
if provided_hash == stored_hash:  # вҶҗ timing leak!

# Safe (from auth.py):
if not hmac.compare_digest(provided_hash, app.app_secret):  # вҶҗ constant time

Q22: What happens when the Settings class validates in production mode?

PROJECTв–ј

The settings.validate() method in config.py performs fail-fast checks at import time:

  • ALLOWED_ORIGINS cannot contain "*" вҖ” prevents accidental open CORS in production
  • AUTH_SERVER_URL must use HTTPS вҖ” prevents token transmission over plain HTTP
  • AUTH_PLATFORM_URL must use HTTPS
  • JWT_PRIVATE_KEY_PEM and JWT_PUBLIC_KEY_PEM must both be present or both absent

This is the fail-fast principle вҖ” it's better to crash on startup with a clear error than silently run with insecure settings and discover the issue in production traffic.

Q23: Explain the IntegrityError handling pattern in _find_or_provision_app_user.

COREв–ј 
def _find_or_provision_app_user(db, app, email):
    target_user = db.query(User).filter(...).first()
    if target_user:
        return target_user
    # Try to create
    target_user = User(email=email, app_id=app.app_id, ...)
    db.add(target_user)
    try:
        db.commit()
    except IntegrityError:
        db.rollback()
        return db.query(User).filter(...).first()  # Race condition resolved
    db.refresh(target_user)
    return target_user

This is the UPSERT / "check-then-create" race condition pattern. Two concurrent requests could both see target_user = None and try to INSERT. The UniqueConstraint causes one to fail with IntegrityError. The handler rolls back and re-queries to get the row the other request created. This is optimistic concurrency control.

Q24: What is the difference between async def and def in FastAPI route handlers?

COREв–ј
  • async def вҶ’ Runs on the main event loop. Must use await for I/O. Blocking calls will stall the entire server.
  • def (sync) вҶ’ FastAPI automatically runs it in a threadpool (run_in_executor). Safe for blocking I/O like SQLAlchemy's synchronous ORM.

This project uses def for route handlers because SQLAlchemy's session (get_db) is synchronous. If they were async def, the synchronous DB calls would block the event loop.

Q25: What are indexes in your User model and why do they exist?

COREв–ј 
class User(Base):
    id = Column(Integer, primary_key=True, index=True)
    email = Column(String, index=True, nullable=False)
    tenant_id = Column(Integer, ForeignKey("tenants.id"), index=True)

Why? The most frequent query pattern is WHERE email = ? AND tenant_id = ? AND app_id = ?. Without indexes on email and tenant_id, PostgreSQL would do a sequential scan вҖ” O(n) for every login. With indexes, it's B-tree lookup вҖ” O(log n).

An ideal improvement would be a composite index on (email, tenant_id, app_id) to make the common 3-column filter a single index seek.
